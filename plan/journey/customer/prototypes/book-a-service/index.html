Protótipo — CUSTOMER: Agendar Serviço

Fluxo de 5 telas. Passos 1, 2 e 4 reutilizam componentes existentes. Telas 0, 3 e a lógica de branching em BookingForm são novos (M13-S02 ou nova story).

  • Tela 0 — Hotsite (ponto de entrada) EXISTS: HeroModule + ServiceListModule + BookingCtaModule
  • Tela 1 — Login com Google GAP: /auth/login — M13-S02
  • Passo 1 — Escolha os serviços EXISTS: ServiceSelectionStep (reuse, sem mudanças)
  • Passo 2 — Calendário e horário EXISTS: AvailabilityCarousel + SlotPicker (reuse)
  • Passo 3 — Revisar pedido GAP: AuthenticatedBookingReviewStep (novo componente)
  • Passo 4 — Confirmar agendamento EXISTS: ConfirmationStep (reuse; endpoint diferente)
O que este protótipo valida:
✅ Fluxo de login → seleção → calendário → revisão → confirmação faz sentido?
✅ Passo 3 sem formulário de dados pessoais é claro para o usuário autenticado?
✅ Resumo de serviço + data no passo 3 (read-only) dá contexto suficiente?
✅ Diferença visual entre o passo 3 do guest (form completo) e do customer (só endereço + foto) é perceptível?
✅ Header com usuário logado em todos os passos pós-login orienta bem?
