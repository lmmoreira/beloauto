GUEST — Responder à Solicitação de Informação

UC-005 A2 · Backend+BFF completos (M08) · Frontend: GAP · Protótipo: Draft

Pré-requisito de código: Antes de implementar, renomear o link no email de /bookings/:id/responder?token= para /bookings/:id/submit-info?token= em send-booking-info-requested-notification.use-case.ts → buildRespondLink(). Esse backend code change vai no mesmo story que cria a página.

Fluxo feliz (happy path)

  • 01 — Formulário de resposta GAP: /bookings/[id]/submit-info — token válido; mostra resumo do agendamento + campo de resposta + upload opcional
  • 01c — Enviando resposta GAP: estado de loading inline — botão desabilitado + spinner durante PATCH
  • 02 — Confirmação GAP: tela de sucesso inline — "Obrigado! Vamos analisar e confirmar em breve."

Estados alternativos (unhappy paths)

  • 01b — Link inválido ou expirado GAP: validação do token na page — token ausente / expirado (TTL 7 dias) / booking já não está INFO_REQUESTED
  • 01d — Erro de validação GAP: campo resposta obrigatório — campo "Sua resposta" vazio ao tentar enviar
  • 01e — Erro de envio GAP: retry após falha de rede — erro de servidor / sem rede; botão "Tentar novamente"

O que este protótipo valida

  • A tela de link inválido é clara o suficiente para um usuário leigo entender o que fazer?
  • O card de "O que precisamos saber" (mensagem do admin) está visível antes do campo de resposta?
  • O upload de fotos é simples o suficiente sem causar fricção para quem não precisa enviar fotos?
  • A tela de sucesso transmite confiança de que a resposta foi recebida e o que acontece a seguir?
  • O link "Criar conta / Entrar" no sucesso gera valor ou é ruído desnecessário nesse momento?
  • A página funciona sem nenhuma autenticação — sem auth bar, sem nav, sem cookie?

Arquitetura (referência para dev-notes.md)

  • BFF endpoint: PATCH /v1/bookings/:id/submit-info/guest?token=<jwt> — @Public()
  • Token payload: { bookingId, tenantId, contactEmail } · HS256 · TTL 7 dias
  • Body: { response: string (min 1), photoUrls?: string[] }
  • Resposta: { bookingId, status: "PENDING", infoSubmittedAt }
  • Sem X-Tenant-Slug header necessário — TenantGuard bypassed por @Public()
