Protótipo — STAFF: Agenda (Fila de Agendamentos + Ciclo de Vida)

Fluxo diário do staff/manager: visualizar a fila de agendamentos do dia, abrir o detalhe de um agendamento e aprovar, rejeitar ou pedir mais informações (UC-003/004/005) — e, para um agendamento já aprovado, marcar como concluído, reagendar ou cancelar (UC-008/009). Mesma página de detalhe (/dashboard/bookings/[id]) em todos os casos — o conteúdo e as ações mudam conforme o status do agendamento. Ator: Staff ou Manager.

  • Tela 0 — Agenda (fila do dia) GAP: /dashboard/bookings — novo page.tsx
  • Tela 1 — Detalhe do agendamento (PENDING) GAP: /dashboard/bookings/[id] — novo page.tsx
  • Tela 2 — Aprovação confirmada GAP: estado de sucesso inline no detalhe
  • Alt A1 — Conflito de horário (slot tomado ao aprovar) GAP: estado de erro inline no detalhe

Estados alternativos (UC-003/004/005)

  • Alt B — Agendamento rejeitado (estado inline) GAP: actionState = 'rejected'
  • Alt C — Info solicitada (estado inline, UC-005) GAP: actionState = 'info-requested'

Formulários de ação (rejeitar / pedir info) são bottom sheets interativos dentro de Tela 1 — não são páginas separadas. Teste no mobile ou redimensione o browser.

Ciclo de vida pós-aprovação (UC-008, UC-009)

Entrada: card na seção "Hoje — confirmados" em Tela 0 → Tela 3.

  • Tela 3 — Detalhe do agendamento (APPROVED) GAP: mesma rota de Tela 1, ramo por status
  • Tela 4 — Marcar concluído (UC-009) GAP: MarkCompleteSheet — preço por linha + fidelidade (A6) + fotos + notas — interativo: editar preços ou usar pontos atualiza o total em tempo real
  • Tela 4b — Conclusão confirmada GAP: estado de sucesso inline — cotado vs. cobrado
  • Tela 5 — Reagendar (UC-008 A1) GAP: reusa o calendário de UC-011
  • Tela 5c — Reagendamento confirmado GAP: estado de sucesso inline — permanece APPROVED

Estados alternativos (UC-008/009)

  • Alt D — Agendamento cancelado pelo staff (estado inline) GAP: actionState = 'cancelled', isBusiness = true
    ↳ Este link mostra o resultado direto, sem clicar no botão. O gatilho real é "Cancelar agendamento" em Tela 3.
  • Alt A2 — Novo horário ficou indisponível ao confirmar GAP: estado de erro inline na tela de reagendar

Bottom sheet de cancelar é interativo dentro de Tela 3 — motivo é opcional (sem mínimo de caracteres, diferente do Rejeitar).

O que este protótipo valida:
✅ Layout da fila (cards) funciona bem em mobile e desktop?
✅ As 3 ações (aprovar / rejeitar / pedir info) cabem bem no card e no detalhe?
✅ Bottom sheet para rejeitar/pedir info/cancelar é o padrão certo em mobile?
✅ Detalhe mostra informações suficientes para o admin decidir sem abrir outra tela?
✅ Validação de motivo de rejeição (mín. 10 chars) é clara?
✅ Erro de conflito de slot (A1) comunica bem o que aconteceu e o que fazer?
✅ O editor de preço por linha (UC-009) é claro sobre cotado vs. cobrado?
✅ A faixa de fidelidade (UC-009 A6) fica clara — staff entende que pode usar pontos como desconto?
✅ O botão "Usar todos" e o campo de pontos são intuitivos? O desconto aparece de forma legível?
✅ Reagendar reusando o calendário de UC-011 é familiar para quem já viu o fluxo de booking?
✅ É claro que cancelar/reagendar/concluir só aparecem em um agendamento já Aprovado?

UCs cobertos: UC-003 (aprovar) · UC-004 (rejeitar) · UC-005 (pedir info) · UC-008 (cancelar/reagendar) · UC-009 (marcar concluído)
