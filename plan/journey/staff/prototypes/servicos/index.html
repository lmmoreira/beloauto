Protótipo — STAFF: Serviços (Catálogo de Serviços)

Fluxo de gestão do catálogo de serviços: listar serviços ativos/inativos, criar um novo serviço (UC-012), editar um serviço existente e desativá-lo (UC-013). Ator: Staff ou Manager.

  • Tela 1 — Lista de serviços (ativos + inativos) GAP: /dashboard/services — novo page.tsx
  • Tela 2 — Criar serviço GAP: /dashboard/services/new (UC-012)
  • Tela 2c — Serviço criado (estado inline) GAP: lista + banner inline (UC-012)
  • Tela 3 — Editar serviço GAP: /dashboard/services/[id]/edit (UC-013)

Estados alternativos (UC-012/013)

  • Alt A — Nome duplicado ao criar (409) GAP: erro inline no campo nome — UC-012 A1
  • Alt B — Confirmação de desativação GAP: /dashboard/services/[id]/deactivate — UC-013 A1
O que este protótipo valida:
✅ A lista distingue claramente serviços ativos de inativos (badges + filtro)?
✅ O filtro Ativos / Inativos / Todos é suficiente ou precisa de busca por nome?
✅ O formulário de criação comunica bem os campos obrigatórios vs. opcionais?
✅ O erro de nome duplicado (409) é claro sobre qual ação corrigir?
✅ O banner inline de "serviço criado" (Tela 2c) é visível o suficiente, sem precisar de um toast?
✅ A desativação como página de confirmação separada é melhor que um bottom sheet?
✅ O aviso de que o preço alterado não afeta agendamentos já confirmados (UC-013 A2) está claro?

UCs cobertos: UC-012 (criar serviço) · UC-013 (editar / desativar serviço)
